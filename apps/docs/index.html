ShadowJS, explained from the beginning

ShadowJS is a TypeScript-first JavaScript framework in this repository. Its main idea is simple: when some data changes, only the code that depends on that exact piece of data should update. The framework does that with signals, a microtask scheduler, a small JSX-to-DOM runtime, a signal-based state layer, and a compiler plugin that turns server imports into RPC calls.

Audience This page is written for someone new to the framework and new to the codebase.
Goal Explain what each part does before expecting you to read the source.
Format You can read this page top to bottom and build the mental model in order.

1. What is ShadowJS

According to the current README, ShadowJS is a framework with four main parts: fine-grained signals, a DOM runtime, state primitives, and a Vite compiler plugin for .server.ts imports.

In practical terms, ShadowJS tries to make UI updates more direct. Instead of re-running a whole component tree and diffing the result, it tracks exactly which computation read which value. Then it updates only the affected computation or DOM binding.

Packages in this repository

  • Core: signals, effects, memos, and the scheduler.
  • Runtime: JSX descriptors, DOM creation, and mounting.
  • State: query, mutation, and store helpers built on signals.
  • Compiler: Babel-based analysis and a Vite plugin for RPC generation.

Short version

  • Signals track data dependencies.
  • The scheduler batches updates.
  • The runtime updates only the exact DOM node that changed.
  • The compiler removes manual client-side fetch wiring for server functions.

2. Signals

A signal is a small container for a value. It gives you two functions: a reader and a writer. 

const [count, setCount] = createSignal(0);

count();                // read the current value
setCount(5);            // write a new value
setCount((v) => v + 1); // functional update

What happens when you read a signal?

If the read happens inside an active effect or memo, the signal records that computation as a subscriber. In the source, this happens through trackDependency().

What happens when you write to a signal?

The signal compares the old value and the new value with Object.is(). If the value really changed, it notifies its subscribers. The important detail is that it does not run those subscribers immediately. It schedules them.

Effects and memos

  • createEffect(): runs a function, tracks its signal reads, and runs it again when one of those reads changes.
  • createMemo(): computes a derived value, caches it, and only recomputes when one of its dependencies changes.
Beginner mental model

Think of a signal as a box that knows who opened it. When the value inside the box changes, it tells only those readers that they need to update.

3. Scheduler

The scheduler is responsible for batching reactive work. In this codebase, it uses queueMicrotask(), not setTimeout().

Why the scheduler exists

Without a scheduler, every signal write would try to run effects immediately. That leads to repeated work and unpredictable timing. ShadowJS instead collects dirty computations in pendingEffects and flushes them after the current synchronous work is done.

What the current implementation does

  1. When a subscriber needs to run, scheduleEffect() marks it dirty and adds it to a set.
  2. If no flush is already queued, the scheduler queues one microtask.
  3. During the flush, memos run before normal effects so derived values stay current.
  4. If more work appears during the flush, another microtask is queued at the end.
Practical result

Multiple writes in the same synchronous block can collapse into one effect run, which keeps updates smaller and more predictable.

4. JSX Runtime

ShadowJS does not use React. Its JSX factory is the function h(tag, props, ...children). That function does not touch the DOM. It only creates a plain descriptor object. 

h("div", { className: "card" }, "Hello");

// becomes a plain object like:
// { tag: "div", props: { className: "card" }, children: ["Hello"] }

From descriptor to real DOM

The runtime function createDOMNode() walks the descriptor tree and builds actual DOM nodes. It handles string tags, function components, arrays of children, and fragments.

Fine-grained DOM updates

The most important beginner concept here is this: when a child or prop is a function, the runtime wraps it in an effect and binds that effect to one specific DOM target.

  • A function child like {name} becomes a targeted text binding.
  • A function prop like title: () => store.label becomes a targeted attribute or property binding.
  • If the current binding is a single text node, the runtime updates that text node directly instead of recreating the parent element.
const [name, setName] = createSignal("Sarthak");

mount(
  () => <div>Hello <span>{name}</span></div>,
  document.body
);

setName("Commander");
Why this matters

In the example above, the runtime only needs to update the text inside the <span>. It does not need to rebuild the whole <div>.

5. State Layer

The state package adds three helpers on top of signals. The goal is to keep local state and async state inside one reactive model instead of introducing a completely separate system.

createQuery()

createQuery(asyncFn, key?) starts in a loading state and then runs the async function. Its signal value always has the shape: { data, error, loading }.

Queries are also registered under a string key so they can be invalidated later.

createMutation()

createMutation(asyncFn, { invalidates }) runs an async action and then refetches any queries registered under the given keys.

createStore()

createStore(initialState) wraps a plain object in a Proxy. Each property gets its own signal entry when it is first accessed or written. 

const posts = createQuery(getPosts, "posts");

const addPost = createMutation(createPost, {
  invalidates: ["posts"]
});

const store = createStore({
  count: 0,
  open: false
});
Simple way to think about it

Queries handle server reads, mutations handle server writes, and stores handle local object state. All three still rely on the same signal mechanics underneath.

6. Compiler / RPC

The compiler package solves one very specific problem: a client file can import a function from a .server.ts file, and the framework turns that import into a client-side RPC stub.

What the analyzer does

The analyzer parses the source with Babel, looks for import declarations whose source matches *.server, and records the imported name, local name, and file path.

What the transform does

The transform removes the original server import and replaces it with generated code that calls fetch("/__rpc/..."). 

export const getPosts = async (...args) => {
  const response = await fetch("/__rpc/posts/getPosts", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(args)
  });

  return await response.json();
};

What the Vite plugin does in development

  • Skips transforming server files themselves.
  • Registers a route for each detected server import.
  • Installs a middleware for /__rpc/* requests.
  • Loads the real server module with ssrLoadModule() and calls the matching export.
Beginner takeaway

You write import { getPosts } from "./posts.server" in client code, but the browser does not receive the real server function. It receives a generated fetch wrapper instead.
